## 🔴 P0 级严重问题详解

### 1. 流程控制缺陷 - 多层循环职责混乱

**问题**：DO 阶段内部包含 5 层循环（主循环、依赖循环、重试循环、恢复循环、资源管理循环），职责过于复杂。同时，**重试决策被分散在三个地方**：

- DO 阶段：executor.py 第 206-222 行，3 次本地重试
- ACT 阶段：actor.py 第 60-66 行，max_retry 次重试决策
- PDCA 宏循环：pdca_loop.py 第 206 行，继续下一轮

**结果**：同一个失败的步骤可能被重试 3+N+1 次（总共最多 9+ 次），且无法区分是哪一层在做决策

**代码示例**：
```python
# executor.py - 第一层重试（不协调）
for attempt in range(max_retries):  # 3 次
    result = self._execute_step(step, context=context)
    if self._quick_check_success(step, result):
        return result  # 立即返回，不通知上层

# pdca_loop.py - 第二层重试（又来了）
for step, exec_result, check_result in zip(...):
    if not check_result["passed"]:
        action = self.actor.decide_action(step, exec_result, check_result)
        if action["action"] == "retry":  # ✗ 又要重试？
            # 继续...
```

### 2. 上下文传播割裂 - 检查阶段信息不足

**问题**：DO 阶段内部生成的上下文（变量替换、步骤输出）只存活在 executor 的局部作用域，不会向上传递给 CHECK 和后续步骤。

**结果**：
- CHECK 阶段无法知道"参数被替换成了什么"，只能看执行结果
- 后续步骤无法获得前序步骤的上下文，导致变量替换失败
- 级联失败风险极高

**代码示例**：
```python
# executor.py 第 126-127 行
context = self._update_context(context, step_id, result)
# 这个 context 在函数返回后就丢弃了！

# pdca_loop.py 第 130 行
check_result = self.checker.check_step_result(
    step=step,
    execution_result=exec_result,
    context=self.executor.memory.get_recent_context()  # ← 只有序列化的观察，不是结构化的上下文
)
```

### 3. 失败处理无序 - DO 和 ACT 重复调整

**问题**：DO 阶段在 `_try_recovery_strategies()` 中会尝试调整参数、切换工具、跳过步骤。但 ACT 阶段又会生成新计划。两个阶段都在"调整"，但没有协调。

**结果**：
- 缓存污染：新计划执行时，executor.execution_cache 仍含有旧结果
- 重复调整：可能尝试了"调整参数"，然后"调整工具"，最后"生成新计划"，浪费资源
- 无法回滚：如果恢复策略失败了，没有机制撤销已做的调整

**代码示例**：
```python
# executor.py 第 240-242 行 - DO 阶段尝试恢复
recovery_result = self._try_recovery_strategies(step, last_error, context)
if recovery_result:
    return recovery_result  # 返回恢复结果

# pdca_loop.py 第 194-201 行 - ACT 阶段又调整计划
current_plan = self.actor.adjust_plan(
    original_plan=current_plan,
    failed_step=step,
    error_info=error_info,
    available_tools=self.planner.get_available_tools()
)
# ✗ 没有清空 executor.execution_cache！
# 新计划会与旧缓存混杂
```

### 4. 收敛性无保证 - 可能无限循环

**问题**：PDCA 循环的终止条件不清晰，多层循环可能陷入无进度的死循环。

**结果**：
- 无法检测"已经无法改进"的状态
- 可能重复调整同一步骤，效果不变
- 达到 max_cycles 后直接停止，可能丢失部分完成的任务

**代码示例**：
```python
# pdca_loop.py 第 73-229 行
while cycle_count < self.max_pdca_cycles:  # 最多 3 次
    # ... PLAN → DO → CHECK → ACT ...
    
    # 但没有监控：
    # - 本轮是否有任何步骤通过检查？
    # - 本轮与上轮相比，已通过步骤数是否增加？
    # - 是否陷入了"调整参数 → 失败 → 调整参数"的循环？

# executor.py 第 92 行
max_iterations = len(plan) * 5  # ← 这是什么逻辑？
# 如果 plan 有 5 步，最多 25 次迭代？
# 与外层的 max_pdca_cycles=3 如何协调？
```

## 📊 为什么"补 Bug"不够？

假设你尝试快速修复：

### 修复方案 A：增加重试次数
```python
max_retries = 5  # 从 3 改成 5
```
❌ **结果**：系统变更慢，仍会失败，只是耗时更长

### 修复方案 B：修复缓存污染
```python
executor.execution_cache.clear()  # 新计划前清空
```
❌ **结果**：后续步骤的变量替换仍然无法正常工作，因为缓存是隔离的，新步骤无法引用前序步骤

### 修复方案 C：修复上下文传递
```python
context_from_do = executor.get_context()  # 从 DO 获取上下文
checker.check_step_result(..., context=context_from_do)  # 传递给 CHECK
```
❌ **结果**：这样虽然能部分解决，但整个流程的架构仍然有根本缺陷（多层循环职责混乱）

### 为什么需要重构

这些问题是**架构级缺陷**，源于：
1. 没有清晰的职责分离（DO 不应做调整决策）
2. 没有统一的上下文管理（各组件各自维护状态）
3. 没有显式的状态机（多层循环无序）
4. 没有收敛监控（无法判断何时停止）

**修修补补只能延缓问题，重构才能根治。**

## ✅ 优化方案总览

### 8 大核心策略

| # | 策略 | 目标 | 预期改进 |
|----|------|------|---------|
| 1️⃣ | 明确职责边界 | DO 只执行，不做重试决策 | 消除多层循环混乱 |
| 2️⃣ | 统一上下文 | PDCAContext 对象贯穿全程 | 所有信息无丢失传递 |
| 3️⃣ | 分层失败处理 | DO 快速重试，ACT 全局决策 | 避免重复调整 |
| 4️⃣ | 显式收敛监控 | ConvergenceMonitor 自动检测 | 防止无限循环 |
| 5️⃣ | 清晰的接口 | 每个组件输入/输出明确定义 | 减少接口不匹配 |
| 6️⃣ | 缓存隔离 | 按循环隔离缓存，自动清理 | 避免缓存污染 |
| 7️⃣ | 异常可见化 | 分层异常体系，完整日志 | 快速诊断问题 |
| 8️⃣ | Fallback 机制 | LLM 失败时降级到本地规则 | 提高容错能力 |

### 收益对标

**定量目标**：
- 循环成功率：30-40% → **85-95%** ↑ 120-220%

- 异常捕获率：60% → **100%** ↑ 67%

**定性目标**：
- ✅ 系统从"不可用" → "生产级可用"
